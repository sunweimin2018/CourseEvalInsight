# 报告生成流程优化需求文档

## 一、背景

当前系统（v1.0）的报告生成流程为"一键式"：用户选择课程/班级/学期后，点击"生成报告"按钮，系统自动解析已上传的课程大纲(Word)、学生信息表(Excel)、成绩表(Excel)，生成包含五个模块的完整报告数据，并在页面上以只读方式预览。用户只能整体导出 Word 或 PDF 格式的完整报告。

此流程存在以下不足：

1. **无法编辑**：解析提取的数据直接用于生成报告，用户无法修改解析错误或不完整的内容
2. **无法分步确认**：五个模块的数据一次性生成，用户无法对单个模块进行确认
3. **无法单独导出**：不能只导出某个模块的 Word 文档用于其他场景
4. **第五模块为空**：课程持续改进方案及措施目前仅为占位文本

## 二、优化目标

将"一键式"报告生成流程改造为"分步式"报告构建流程，使用户能够：

1. 分别生成五个模块的数据
2. 在网页表单中对每个模块的字段进行编辑和定制
3. 分别确认每个模块的数据
4. 分别导出每个模块为独立的 Word 文档
5. 在所有模块确认后，合并导出为最终的完整 Word 和 PDF 报告

## 三、详细设计

### 3.1 整体流程

```
上传文件 → 选择课程/班级/学期 → 进入报告构建页面
                                    │
    ┌───────────────────────────────┼───────────────────────────────┐
    │                               │                               │
    ▼                               ▼                               ▼
[模块1: 课程基本信息表]    [模块2: 课程目标]    [模块3: 课程评价标准]
    │                               │                               │
    ├─ 生成数据                     ├─ 生成数据                     ├─ 生成数据
    ├─ 在表单中编辑                 ├─ 在文本框中编辑               ├─ 在编辑器中编辑
    ├─ 保存草稿 / 确认              ├─ 保存草稿 / 确认              ├─ 保存草稿 / 确认
    └─ 导出单独Word                 └─ 导出单独Word                 └─ 导出单独Word
                    │                               │
    ┌───────────────┤                               ├───────────────┐
    │               │                               │               │
    ▼               ▼                               ▼               ▼
[模块4: 课程评价结果]                    [模块5: 持续改进方案]
    │                                               │
    ├─ 生成数据                                     ├─ 生成数据
    ├─ 在表单中编辑                                 ├─ 在文本框中编辑
    ├─ 保存草稿 / 确认                              ├─ 保存草稿 / 确认
    └─ 导出单独Word                                 └─ 导出单独Word
                    │                               │
                    └───────────┬───────────────────┘
                                ▼
                    [合并导出完整报告]
                        ├─ Word 格式
                        └─ PDF 格式
```

### 3.2 页面布局

采用左右分栏布局：

- **左侧（窄列，约 220px）**：垂直步骤指示器
  - 步骤1：课程基本信息表
  - 步骤2：课程目标
  - 步骤3：课程评价标准
  - 步骤4：课程评价结果
  - 步骤5：课程持续改进方案及措施
  - 最终步骤：合并导出
  - 每步显示状态指示：灰色(待生成) / 蓝色(草稿) / 绿色(已确认) / 红色(失败)

- **右侧（宽列）**：
  - 顶部：课程/班级/学期选择器 + 文件状态面板
  - 中部：当前步骤对应的模块编辑器
  - 底部：操作按钮（重新生成 / 保存草稿 / 确认并导出Word）

### 3.3 模块一：课程基本信息表

**数据来源**：课程大纲 Word 文档（解析字段）+ 学生信息 Excel（统计人数/性别/班级）

**可编辑字段**：
| 字段名 | 控件类型 | 说明 |
|--------|---------|------|
| 课程名称 | 单行文本框 | 从大纲解析 |
| 课程编号 | 单行文本框 | 从大纲解析 |
| 授课班级 | 单行文本框 | 从大纲解析 |
| 修课人数 | 单行文本框 | 自动统计，可手动修改 |
| 课序号 | 单行文本框 | 从大纲解析 |
| 学时数 | 单行文本框 | 从大纲解析 |
| 学分 | 单行文本框 | 从大纲解析 |
| 选用教材 | 单行文本框 | 从大纲解析 |
| 开课院系 | 单行文本框 | 从大纲解析 |
| 授课教师 | 单行文本框 | 从大纲解析 |
| 课程性质 | 单行文本框 | 从大纲解析 |
| 课程类型 | 单行文本框 | 从大纲解析 |

**只读统计信息**：学生总人数、男生人数、女生人数、班级分布列表

### 3.4 模块二：课程目标

**数据来源**：课程大纲 Word 文档（大段文本，从"课程目标"标题下提取）

**编辑控件**：多行文本框（textarea），支持长文本编辑

### 3.5 模块三：课程评价标准

**数据来源**：课程大纲 Word 文档（段落 + 表格混合内容）

**编辑控件**：
- 段落块：多行文本框
- 表格块：可编辑表格（支持双击单元格编辑）
- 保持段落与表格的原始顺序

### 3.6 模块四：课程评价结果

**数据来源**：成绩 Excel 文件（自动分析各成绩列）

**可编辑内容**：
- 每列成绩的统计指标：参考人数、缺考人数、最高分、最低分、平均分、中位数、标准差、及格率
- 成绩分布表（优秀/良好/中等/及格/不及格各段人数）
- 图表预览：柱状图（成绩分布）、饼图（及格率）根据编辑后的数据实时更新

### 3.7 模块五：课程持续改进方案及措施

**数据来源**：默认提供结构化模板，用户自行填写

**模板结构**：
1. 课程目标达成情况
2. 教学过程中存在的问题
3. 持续改进措施
4. 责任人及时间节点

**编辑控件**：多行文本框（textarea）

### 3.8 合并导出

当所有五个模块均为"已确认"状态时，允许合并导出。

- 将五个模块按顺序合并到一个 Word 文档中
- 合并后的文档可导出为 .docx 和 .pdf 两种格式
- 若存在未确认模块，提示用户先完成确认

## 四、技术实现要点

### 4.1 后端

- 在 `ReportRecord` 模型中增加 `module_1_status` ~ `module_5_status` 字段，追踪每模块状态
- 拆分 `report_engine.py` 为独立的模块生成函数
- 新增 5 个模块级 API 端点（生成、更新、导出Word）+ 1 个合并端点
- 单个模块的 Word 导出共享文档构建工具函数

### 4.2 前端

- Vue 3 + Element Plus 实现步骤式交互
- 5 个独立的模块编辑器组件，每个提供适合其数据类型的编辑界面
- Pinia 暂不增加新的 store，状态由报告构建页面本地管理
- ECharts 图表在模块四中根据编辑后的数据实时更新

## 五、数据状态流转

每个模块的状态机：

```
待生成(pending) ──[生成]──▶ 草稿(draft) ──[确认]──▶ 已确认(confirmed)
                              │                        │
                              └──[重新生成]──▶ 草稿    └──[重新生成]──▶ 草稿
```

## 六、兼容性说明

- 保留现有 `/report/generate` 快速预览页面，不对已有功能进行破坏性修改
- 新的步骤式构建页面通过新路由 `/report/builder` 访问
- 现有的 `ReportRecord` 表通过 migration 新增字段，已有数据的模块状态默认为 `pending`
